The Solution

Four files. Zero dependencies.

PM
Product Manager
/feature-spec
Defines WHAT and WHY. Socratic questioning, acceptance criteria, impact analysis.
AR
Architect
/feature-plan
Defines HOW and WHERE. Granular tasks with adversarial subagent review.
QA
QA Engineer
/tdd
RED-GREEN-REFACTOR. Real E2E tests against real services, not mocks.
OC
Orchestrator
feature-workflow
Auto-reminds Claude to follow the flow. Skips for trivial changes.

No CLI tools. No npm packages. Just copy into .claude/

The Flow

From idea to shipped code

/feature-spec PM asks clarifying questions → produces spec with requirements & acceptance criteria
/feature-plan Architect breaks into 2-5 min tasks → adversarial subagent reviews for gaps
/tdd Write failing test (RED) → minimum code (GREEN) → refactor. Real E2E, not mocks.
Ship it Execute remaining tasks → tests pass → review → commit

In Practice

Real example: email notifications

SPEC "I want email notifications when an order completes"
PM asks: Who receives? What triggers completion? Guest checkout?
PLAN Task 1: Create template in templates/order-complete.ts
Task 2: Add trigger in api/orders/complete/route.ts
Task 3: E2E test — POST triggers real email
RED Write test: POST /api/orders/complete → email sent to buyer with order details
GREEN Implement handler. Test passes. Refactor. Ship.
06 / 08
Why This Way

Key design decisions

Why skills, not agents? Claude already knows all the roles. Skills run in-context, no orchestration overhead. You control when to switch hats.
Why adversarial review? A subagent critiques your plan for completeness, architecture, ordering, risks, and scope before you see it. Catches blind spots.
Why real tests over mocks? Mocks test your assumptions. Real E2E tests catch what mocks miss: broken routes, auth issues, race conditions, missing env vars.
Why lightweight specs? ~1 page with clear acceptance criteria is enough for solo devs. The overhead of multi-artifact workflows isn't worth it.
